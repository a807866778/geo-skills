# 品牌知识库构建器 (geo-knowledge-base)

## 概述

本技能用于从官方来源系统化构建品牌知识库。品牌知识库是所有GEO优化工作的信息基座——它确保品牌在AI生成回答中被引用时，信息准确、完整且可验证。本知识库覆盖品牌事实、产品/服务、FAQ和权威证据四大板块，每条信息必须标注来源。

## 核心原则

1. **来源可追溯**：每条事实必须标注可追溯的官方来源
2. **未经核实即标记**：无法验证的信息标记为【未验证】而非省略或编造
3. **结构化优先**：知识以分类、卡片、数据库的形式组织，便于AI检索和提取
4. **持续增量更新**：知识库设计为可增量更新的文档系统
5. **事实与主张分离**：严格区分客观事实（可验证的）与主观主张（营销性的）

## 知识库四大板块

| 板块 | 核心内容 | 信息载体 | 最低要求 |
|------|----------|----------|----------|
| A. 品牌事实卡片 | 品牌基础信息、定位、历史、规模 | 事实卡片（Structured Fact Cards） | 15张核心卡片 |
| B. 产品/服务目录 | 产品规格、服务参数、定价、适用场景 | 结构化目录（Spec Sheet） | 覆盖所有核心产品/服务 |
| C. FAQ数据库 | 高频问题与标准答案 | FAQ条目（Q&A Pair） | 50+ 条 |
| D. 权威证据目录 | 认证、奖项、专利、媒体覆盖 | 证据卡片（Evidence Card） | 10+ 条证据 |

## 信息来源规范

### 优先级分级（从高到低）

| 优先级 | 来源类型 | 示例 | 引用格式 |
|--------|----------|------|----------|
| L1 | 官方渠道 | 品牌官网、官方公众号、企业年报 | `[来源: 官网URL, 访问日期]` |
| L2 | 官方文档 | 白皮书、产品手册、技术规格书、新闻稿 | `[来源: 文档名称, 版本/日期]` |
| L3 | 官方资质 | 营业执照、认证证书扫描件、检测报告、专利证书 | `[来源: 资质编号, 颁发机构, 有效期]` |
| L4 | 权威第三方 | 政府公示、行业数据库（天眼查/企查查）、权威媒体报道 | `[来源: 平台/媒体名称, URL, 发布日期]` |
| L5 | 用户提供 | 用户在对话中直接提供的信息，尚未在官方来源核实 | `[来源: 用户提供, 待核验]` |

### 禁止使用的来源

- 不可靠的UGC平台（如未经认证的知乎回答、个人博客）
- 竞争对手网站（可能包含不准确信息）
- AI生成的内容（不可作为事实来源）
- 已过期的信息（除非标注历史记录用途）

## 知识库构建工作流

### 第一阶段：来源扫描与信息采集

**执行指令**：

```
请系统化采集以下官方来源的品牌信息：
1. 品牌官网（所有页面的文本内容）
2. 官方白皮书/行业报告
3. 产品文档/规格书
4. 官方新闻稿/公告
5. 官方社交媒体账号
6. 品牌方提供的资质文件
7. 第三方权威平台公示信息（天眼查、行业数据库等）

对每条采集的信息，记录：
- 原始文本/数据
- 来源URL/文档名称
- 采集时间戳
- 信息类型分类
```

### 第二阶段：信息分类与实体归属

将采集到的信息按实体类型分类：

```
实体分类标准：

1. 品牌实体 (Brand Entity)
   - 品牌名称、注册名称
   - 成立时间、注册资本
   - 品牌使命/愿景/价值观
   - 品牌里程碑/发展历程
   - 组织架构、母子公司关系

2. 产品/服务实体 (Product/Service Entity)
   - 产品名称、型号
   - 核心功能/参数
   - 适用人群/场景
   - 价格/套餐
   - 版本历史
   - 对比竞品的差异化特性

3. 人物实体 (Person Entity)
   - 创始人/核心团队的教育背景、职业经历
   - 专家团队资质
   - 顾问委员会

4. 地点实体 (Location Entity)
   - 总部地址
   - 分支机构/门店/服务网点
   - 服务覆盖区域

5. 资质实体 (Credential Entity)
   - 行业许可、经营资质
   - 认证体系（ISO、CE、FDA等）
   - 检测报告
   - 知识产权（专利、商标、软著）

6. 荣誉实体 (Achievement Entity)
   - 获奖记录
   - 行业排名
   - 媒体报道
   - 客户/用户数量里程碑

7. 内容实体 (Content Entity)
   - 方法论/理论框架
   - 自有研究/数据报告
   - 案例研究
```

### 第五阶段：FAQ数据库生成

#### FAQ生成规则

FAQ问题来源优先级：
1. 从品牌官方FAQ页面提取
2. 从产品文档/说明书中提取用户常见问题
3. 从客服记录/用户反馈中总结高频问题
4. 从行业通用问题推导（按照 `geo-question-discovery` 的7维框架）
5. 从竞品FAQ中发现遗漏问题

#### FAQ条目模板

```markdown

| 字段 | 内容 |
|------|------|
| **问题** | [用户自然语言问题] |
```

## 信息质量规则

### 必须遵守的规则

1. **每条事实必须有来源**：没有来源的事实不得进入知识库
2. **无法验证的信息必须标注**：标记为【未验证】【用户提供待核验】而非删除
3. **主观主张必须注明**：区分"客观事实"和"品牌主观主张"
4. **时效性必须追踪**：每条信息标注采集日期，有有效期的标注截止日期
5. **不确定性必须量化**：使用"约""估计""区间"等措辞代替精确但不可靠的数字
6. **禁止编造**：不得虚构任何数据、资质、案例、客户

### 事实置信度标注标准

| 置信度 | 标注 | 条件 |
|--------|------|------|
| 已确认 | ✅ | 官方来源可查，信息明确且无歧义 |
| 高置信 | 🟢 | 官方来源暗示或可推导，但未明确陈述 |
| 待核验 | 🟡 | 用户提供或非官方来源，尚未在官方渠道确认 |
| 近似值 | 🔵 | 数据来自估算或行业推测，非精确数值 |
| 存疑 | 🔴 | 来源之间存在矛盾或信息与常识不符 |
| 已过期 | ⏰ | 信息曾为真但已超过有效期 |

### 信息冲突处理

当不同来源对同一事实给出不同信息时：
1. 记录所有版本
2. 标注每个版本的来源和采集时间
3. 标记为【信息冲突，待解决】
4. 优先采信L1来源（官方）的信息作为主版本

## 知识库维护规则

### 增量更新流程

```
1. 用户提供新信息 → 判断是否与现有知识冲突
2. 无冲突 → 新增为补充卡片，更新关联条目
3. 有冲突 → 按冲突处理规则记录，等待用户确认
4. 信息过期 → 保留历史版本，新增更新版本卡片
5. 每季度全面复核 → 检查链接有效性、信息时效性
```

### 版本管理

```
- 知识库主版本号: v1.0, v1.1, v2.0
- 每次重大更新（新增>10%条目）提升次版本号
- 每次修正错误信息提升修订号（v1.0.1）
- 保留版本变更日志
```

## 跨平台使用说明

本技能为纯文本指令集，可在任何支持Markdown的AI代理平台使用。使用时：
- 将本SKILL.md的内容作为系统指令加载
- 提供品牌官方来源（网址、文档、资料）
- 按六阶段工作流系统构建知识库
- 输出标准化品牌知识库文档（含事实卡片、产品目录、FAQ数据库、证据目录）
- 所有模板、规则、标注体系均内嵌于文档中，无需外部依赖
